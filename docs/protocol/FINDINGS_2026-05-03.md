# Protocol Findings — 2026-05-03 (opcode-structure pass)

This document captures the structural rules + new packet
identifications discovered when systematically analyzing the
opcode space. See [`OPCODE_STRUCTURE.md`](OPCODE_STRUCTURE.md)
for the full reference; this doc is the change log.

## Newly identified `0x03/0x1f` tag packets

`0x03/0x1f` is the protocol's general-purpose state-event channel.
Sub-tag analysis (`tools/subtag_analysis.py`) on 13 captures
(104,005 packets) revealed the following NEW packet roles:

### Tag `0x01` — weapon target / aim packet

**Direction:** C→S (98% of obs)  
**Size:** 19 bytes fixed  
**Body shape:**

```
01 00 01 23 01 [target_entity_id LE4] [aim/hit data ~10B]
```

Verified by inspection: at LOOT marker (t=1093.09), the just-killed
crawler had entity_id `0x03bd`. The C→S `01 00 01 23 01 bd 03 00 00 …`
packets at t=1065-1071 match exactly. So this is the
"player-shooting-at-target" packet, with the target entity ID in
bytes 5-8.

The frequency (~0.5 Hz when actively engaged, sparse when
idle) is consistent with rifle-fire cadence rather than
continuous aim-tracking.

**Implication:** the long-standing "what packet does the player
fire?" question is answered. PvE combat uses
`UDP C→S 0x03/0x1f tag=0x01 sub=0x23` with target ID embedded.

### Tag `0x1a` — dialogue/mission state machine

**Direction:** mixed  
**Size:** 4-22B (variable)  
**Body shape:**

| Direction | Size | Body | Meaning |
|---|---:|---|---|
| C→S | 4 | `01 00 1a 01` | "select dialogue option 1" |
| S→C | 6 | `01 00 1a [option_id] 00 00` | "next dialogue node is option_id" |
| S→C | 10 | `01 00 1a 00 00 01 00 00 80 3f` | dialogue init + 1.0 float (progress?) |

**Verified marker correlations:** TALKED_NPC1_CLICKDIALOG2,
TALKED_NPC2_CLICKDIALOG13, OUTSIDE_AREAM5_TALKING_GEORDI_*,
OUTSIDE_AREAM5_TALKING_EZRA_*, OUTSIDE_AREAM5_MISSION_COMPLETE
(5 obs).

This is the **dialogue navigation channel** — bytes 3-4 encode the
selected dialogue tree node. Mission-complete events fire the
same channel because mission completion is just "click dialogue
option marked 'complete'".

### Tag `0x2a` — mission grant / state update

**Direction:** S→C (100%)  
**Size:** 58-64 bytes  
**Body shape:**

```
01 00 2a [version 1B] 01 00 00 [uid LE3] 12 [mission_id ASCII 11B + nulls] [state ~30B]
```

**Decoded mission IDs from the capture:**

| ASCII string | NPC marker |
|---|---|
| `mc5_lorcan` | OUTSIDE_AREAM5_TALK_NPCLORCAN |
| `mc5_ezra` | OUTSIDE_AREAM5_TALKING_EZRA_* |
| `mc5_tasha` | OUTSIDE_AREAM5_TALKING_TASHA_* |
| `mc5_geordi` | OUTSIDE_AREAM5_TALKING_GEORDI_* |

**Convention:** mission IDs are `<area>_<npc>` ASCII strings,
null-padded to 16 bytes (11 chars + 5 nulls observed). The
`mc5_` prefix names the area (Military Command 5 / AreaMC5
in markers).

This is the **server-side quest log update** — the server pushes
the current mission state to the client every ~tens of seconds
during an active mission.

### Tag `0x26` — entity inventory / loot listing

**Direction:** S→C (100%)  
**Size:** 10-821 bytes (variable, content-dependent)  
**Body shape:**

```
01 00 26 [entity_id LE4] [item count?] [TLV item list]
```

Verified at LOOT marker (entity_id `0x03bd` matches the crawler).
Same channel used for vendor listings (NPC_VENDOR_OPEN /
SELL markers). 

The size variability tracks the number of items:
- 10B = empty inventory or single-item entity
- 821B = full vendor stock

### Tag `0x1e` — inventory slot action

**Direction:** C→S  
**Size:** 11 bytes fixed  
**Body shape:**

```
01 00 1e [op 1B] [src 2B] [dst 2B] [count 2B] 01 00
```

Fires during EQUIPING markers. Works together with the 12-byte
`UDP C→S 0x00` inventory channel — `0x1e` is the high-level
"item action" notification while `0x00` is the slot-level put.

## TCP opcode structure (verified)

The 16-bit TCP opcode space has clean human-designed segmentation:

| High byte | Subsystem | Direction tendency |
|---|---|---|
| `0x80xx` | Connection handshake (3-way) | both |
| `0x83xx` | Game session state | almost always S→C |
| `0x84xx` | Account / character | always C→S |
| `0x87xx` | Gamedata / config sync | request/response |
| `0xa0xx` | Session-ready | request/response |

**Request/response pairing rule:**
- C→S `0x84xx` → S→C `0x83xx` (with related low byte)
- e.g. `0x8480` Auth → `0x8381` AuthAck (low byte `0x80` ↔ `0x81`)
- e.g. `0x8482` GetCharList → `0x8385` CharList (`0x82` ↔ `0x85`)

**Opcode reuse via discriminator body:**
- `0x8482` is one opcode that serves both `GetCharList` (30B body)
  and `CharCreate` (39B preview / 110B commit). Body bytes 6-7
  discriminate.

## UDP opcode structure (verified)

UDP opcodes are 1 byte, organized in 16-byte ranges:

| Range | Subsystem |
|---|---|
| `0x00`-`0x0f` | Connection / sync / wrappers |
| `0x10`-`0x1f` | Entity & state stream |
| `0x20`-`0x2f` | World events |
| `0x30`-`0x3f` | Player metadata / inventory |
| `0x40`-`0xff` | Sparse — runtime / cipher echoes |

**Channel duality:** for most opcodes both raw `0xN` and
reliable `0x03/0xN` exist with the **same logical content**;
the choice is reliability.

**Wrapper opcodes** (don't carry data directly):
- `0x03` reliable
- `0x07` multipart (only inside `0x03/0x07`)
- `0x13` gamedata batching (legacy; in retail, sub-packets
  travel directly under `0x03/...`)
- `0x1f` GamePackets (sub-tagged event channel)
- `0x25` (sub-tagged inside `0x1f` for transactions)
- `0x2b` CityCom DCB (RPC method strings)
- `0x32` NPC dialogue (option strings)

## Inventory channel — `0x00` 12B fixed

The 12-byte inventory channel uses a tight 10-op enum
(`0x00`-`0x09`):

| op | Action |
|---|---|
| 0 | loot pickup (corpse → inventory) |
| 1 | drag-from slot |
| 2 | drag-to slot |
| 3 | swap / move |
| 4 | use / consume |
| 5 | drop / recycle |
| 6 | drop variant |
| 7 | take from container |
| 8 | put into container |
| 9 | equip / unequip |

Predicted unobserved: 0x0a (stack split), 0x0b (sort), 0x0c-0x0d
(container open/close).

## Cipher echo / mystery packets

Several rare 1-byte opcodes (`0x05`, `0x06`, `0x25`, `0x31`) appear
exactly once in the entire 13-capture corpus. Their bodies are
1 byte each (just the opcode). These are likely **cipher state
markers** emitted during the very first UDP packet of a session,
NOT real protocol packets.

The 10-byte rare opcodes (`0x0d`, `0x0f`, `0x11`, `0x1d`) follow
a consistent pattern:

```
[opcode] 00 03 [LE2] 1f 01 00 [tag] [tag2]
```

This is structurally identical to a `0x13` gamedata wrapper
carrying a single `0x1f` sub-packet. The opcode byte appears to
select an event class — analogous to how `0x1b` is "the
single-sub entity-update wrapper". Each rare opcode is its own
event class; the sample diversity is too low to determine which.
